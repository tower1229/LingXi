# 项目文档写作风格指南

本文档提取自项目 README.md 的语言风格和文字审美取向，用于指导文档编写时保持一致的风格。**此指南专注于语言风格和审美取向，不对文档结构或内容做具体限制。**

## 语言风格

### 1. 简洁有力

- **短句为主**：避免冗长复杂的句子结构
- **直接表达**：开门见山，不绕弯子
- **精炼用词**：每个词都有明确作用，避免冗余

**示例对比**：

```
❌ 不够好：这个功能的设计理念是为了让用户在使用过程中能够更加便捷地完成他们的工作任务。
✅ 更好：这个功能让用户更便捷地完成工作。
```

### 2. 专业而温暖

- **中文表达**：使用中文，但保持专业术语的准确性
- **有温度**：避免过度技术化，保持可读性和亲和力
- **适度专业化**：在专业和易懂之间找到平衡

**示例对比**：

```
❌ 过于技术化：通过 API 端点实现数据交互，采用 RESTful 架构模式进行资源访问。
✅ 更好：通过 API 接口进行数据交互，使用 RESTful 标准。
```

### 3. 实用导向

- **强调"做什么"和"如何使用"**：优先说明功能和使用方法
- **提供具体示例**：用代码块、命令示例等展示实际用法
- **避免抽象描述**：用具体场景和例子说明概念

**示例**：

```markdown
### 快速开始

在 Cursor 输入：
```

/req <一句话需求描述>

```

```

## 视觉呈现

### 1. 格式规范

- **加粗突出**：使用 `**加粗**` 突出关键信息、功能名称、重要概念
- **代码块**：使用代码块展示命令、配置、代码示例
- **引用块**：使用 `>` 引用块强调注意事项、警告、重要提示
- **表格组织**：使用表格组织结构化信息（属性、功能列表、对比等）
- **列表使用**：使用有序/无序列表组织要点

**示例**：

```markdown
- **极简入口**：
- **可伸缩工作流**：
- **质量资产化**：持续沉淀过程产物，实时提取质量标准

> **注意**：安装脚本会在当前项目目录创建 `.cursor/` 和 `.cursor/.lingxi/` 目录结构。
```

### 2. 层次分明

- 使用清晰的标题层级（H1 → H2 → H3）
- 使用分隔线区分不同主题区块（如需要）
- 使用空行增加可读性

### 3. 可读性

- **视觉引导**：通过格式和排版引导读者注意力
- **信息密度适中**：既不过于稀疏，也不过于密集
- **易于扫描**：使用列表、表格、加粗等让内容易于快速浏览

## 文字审美取向

### 1. 精炼原则

- **少即是多**：用最少的文字传达最多的信息
- **避免重复**：不在不同章节重复相同内容
- **去除冗余**：删除不必要的修饰词和解释

**示例对比**：

```
❌ 冗余：这个功能非常强大，它能够帮助我们快速地完成很多不同的任务。
✅ 精炼：这个功能能快速完成多种任务。
```

### 2. 专业原则

- **术语准确**：使用准确的技术术语，但避免过度专业化
- **表达规范**：遵循中文表达规范，保持语法正确
- **逻辑清晰**：内容组织符合逻辑，易于理解

### 3. 可读性原则

- **视觉引导**：通过格式和排版引导读者注意力
- **信息密度适中**：既不过于稀疏，也不过于密集
- **易于扫描**：使用列表、表格、加粗等让内容易于快速浏览

## 应用指南

### 编写文档时的风格要求

1. **语言风格**：

   - 使用短句，直接表达
   - 保持专业但易懂
   - 提供具体示例

2. **视觉呈现**：

   - 使用加粗、代码块、表格等增强可读性
   - 保持层次清晰
   - 易于扫描

3. **文字审美**：
   - 精炼表达，避免冗余
   - 专业准确，逻辑清晰
   - 信息密度适中

### 风格检查清单

- [ ] 语言是否简洁有力（短句、直接表达）
- [ ] 是否使用了适当的格式工具（加粗、代码块、表格等）
- [ ] 是否提供了具体示例（如适用）
- [ ] 是否易于扫描和理解
- [ ] 是否避免了冗余和重复
- [ ] 表达是否专业而温暖（不过度技术化）

## 注意事项

1. **风格优先，结构灵活**：此指南专注于语言风格和审美取向，不对文档结构或内容组织方式做具体限制
2. **精炼原则**：避免冗余，保持简洁有力
3. **可读性**：使用格式工具增强可读性，但不要过度使用
4. **适度专业化**：在专业和易懂之间找到平衡
