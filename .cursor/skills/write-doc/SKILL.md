# Write Doc

## Overview

此 Skill 帮助编写和修改项目文档，确保文档的语言风格和文字审美取向与项目 README.md 保持一致。通过参考结构化的写作风格指南，指导文档的编写和修改过程。

## 工作流程

### 1. 理解任务

在开始编写/修改文档前，明确：

- **文档类型**：README、技术文档、使用指南、API 文档等
- **目标受众**：用户、维护者、开发者等
- **文档目的**：介绍项目、使用指南、技术细节等
- **现有内容**：是否需要修改现有文档，还是创建新文档

### 2. 加载风格指南

在编写前，必须读取 `references/writing-style.md` 了解项目的语言风格和审美取向：

- 语言风格特征（简洁有力、专业而温暖、实用导向）
- 视觉呈现规范（格式、层次、可读性）
- 文字审美取向（精炼、专业、可读性）

**注意**：风格指南专注于语言风格和审美取向，不限制文档结构或内容组织方式。

### 3. 分析现有文档（如适用）

如果修改现有文档，先分析：

- 当前文档的语言风格
- 与风格指南的差异
- 需要改进的地方（语言表达、格式、可读性）
- 需要保留的内容和结构

### 4. 编写/修改文档

按照风格指南编写或修改文档，重点关注：

#### 4.1 语言风格

- **简洁表达**：使用短句，直接表达核心内容
- **专业而温暖**：保持专业但不失亲和力
- **实用导向**：提供具体示例和使用方法

#### 4.2 视觉呈现

- **格式规范**：使用加粗、代码块、表格、列表等增强可读性
- **层次分明**：使用清晰的标题层级和适当的空行
- **易于扫描**：使用格式工具让内容易于快速浏览

#### 4.3 文字审美

- **精炼原则**：避免冗余，保持简洁有力
- **专业准确**：使用准确的技术术语，但不过度专业化
- **可读性**：信息密度适中，易于理解

### 5. 风格检查

完成编写后，使用检查清单验证：

- [ ] 语言是否简洁有力（短句、直接表达）
- [ ] 是否使用了适当的格式工具（加粗、代码块、表格等）
- [ ] 是否提供了具体示例（如适用）
- [ ] 是否易于扫描和理解
- [ ] 是否避免了冗余和重复
- [ ] 表达是否专业而温暖（不过度技术化）

### 6. 输出文档

- 直接写入目标文件
- 遵循"静默成功"原则：写入成功不输出确认信息，失败才输出错误

## 使用场景

### 场景 1：创建新文档

用户："帮我写一个 API 文档" 或 "创建一个使用指南"

**流程**：

1. 明确文档类型和目标受众
2. 根据文档类型和目标设计文档结构（不受风格指南限制）
3. 加载风格指南，了解语言风格和审美取向
4. 按照风格指南编写内容（语言风格、格式、可读性）
5. 进行风格检查
6. 输出文档

### 场景 2：修改现有文档

用户："优化这个 README" 或 "改进文档的可读性"

**流程**：

1. 读取现有文档
2. 加载风格指南
3. 分析现有文档的语言风格与风格指南的差异
4. 保留现有结构和内容，按风格指南优化语言表达和格式
5. 进行风格检查
6. 输出修改后的文档

### 场景 3：确保风格一致性

用户："检查这些文档的风格是否一致" 或 "让文档风格与 README 保持一致"

**流程**：

1. 读取目标文档
2. 加载风格指南
3. 对比分析语言风格和审美取向的差异
4. 提出改进建议或直接修改（只修改风格，不改变结构）
5. 输出结果

## 风格指南参考

详细的写作风格指南请参考 `references/writing-style.md`，包括：

- 语言风格（简洁有力、专业而温暖、实用导向）
- 视觉呈现规范（格式、层次、可读性）
- 文字审美取向（精炼、专业、可读性）
- 风格示例
- 应用指南和检查清单

## 注意事项

1. **风格优先，结构灵活**：此 Skill 专注于语言风格和审美取向，不对文档结构、内容组织方式或具体内容做限制
2. **精炼原则**：避免冗余，保持简洁有力
3. **可读性**：使用格式工具增强可读性，但不要过度使用
4. **适度专业化**：在专业和易懂之间找到平衡
5. **静默成功**：文档写入成功时不输出确认信息
