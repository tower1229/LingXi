# Init 阶段触发场景

本文档定义 init 阶段的经验候选触发场景。

## 触发场景列表

| 触发场景 | 描述 | 典型信号 |
|---------|------|---------|
| **技术栈选择理由** | 为什么选择某个技术栈/框架/库 | "选择 React 因为..."、"使用 Hono 因为..." |
| **架构决策** | 为什么选择某个架构模式/设计模式 | "采用依赖注入因为..."、"使用工厂函数因为..." |
| **开发规范取舍** | 为什么采用某个开发规范 | "使用 TypeScript strict 因为..."、"命名规范选择 kebab-case 因为..." |
| **项目结构决策** | 为什么采用某个项目结构 | "目录结构设计为..."、"入口文件选择..." |
| **常见坑点** | 开发中常遇到的问题或陷阱 | "注意...容易出错"、"避免..." |
| **业务规则** | 业务层面的约束和规则 | "配额限制..."、"权限控制..." |

## 经验提取要点

**技术栈选择理由**的提取要点：

- **技术栈组合**：为什么选择这个技术栈组合？
- **取舍理由**：考虑了哪些其他技术栈？为什么最终选择这个？
- **判断依据**：基于什么做出技术栈选择（性能、生态、团队熟悉度等）？

**架构决策**的提取要点：

- **架构模式**：为什么选择某个架构模式（依赖注入、工厂函数、无全局单例等）？
- **设计原则**：体现的设计原则和价值取向（可测试性、可维护性、可扩展性等）？
- **取舍理由**：拒绝了哪些其他架构方案？为什么？

**开发规范取舍**的提取要点：

- **规范选择**：为什么采用某个开发规范（代码规范、命名规范、测试规范等）？
- **价值取向**：体现了什么价值取向（可读性、一致性、团队协作等）？
- **约束理由**：为什么设置某个约束条件？

## 典型信号示例

- **技术栈选择理由**：用户说"选择 React 19 因为..."、"使用 Hono 因为..."
- **架构决策**：用户说"采用依赖注入因为..."、"使用工厂函数因为..."
- **开发规范取舍**：用户说"使用 TypeScript strict 因为..."、"命名规范选择 kebab-case 因为..."
- **项目结构决策**：用户说"目录结构设计为..."、"入口文件选择..."
- **常见坑点**：用户说"注意...容易出错"、"避免..."
- **业务规则**：用户说"配额限制..."、"权限控制..."

## 注意事项

- 重点识别**判断和取舍**，而非单纯的技术栈列表
- 关注**为什么**，而非**是什么**
- 识别**可复用的决策模式**，而非项目特定的细节
