# 候选评估机制设计

## 概述

候选评估机制是知识沉淀流程中的核心环节，统一处理所有 EXP-CANDIDATE 的质量评估和分类决策。它替代了原来分散的"噪音过滤器"、"成长过滤器"和"沉淀分流"，提供了一个统一、严谨的评估框架。

## 设计原则

1. **统一评估**：所有候选都通过同一套评估标准，避免重复和冲突
2. **多维评估**：从多个维度评估候选价值，而非单一问题
3. **明确分类**：根据评估结果明确分类，而非模糊判断
4. **可解释性**：每个评估结果都有明确理由，支持人工复核
5. **保护品味**：边界模糊的候选仍由人最终判断

## 评估维度

### 1. 结构完整性（Structural Integrity）

评估候选是否包含完整的判断结构。

**评估项**：
- ✅ 包含 `decision` 字段（判断了什么）
- ✅ 包含 `alternatives` 字段（拒绝了什么，至少 1 个）
- ✅ 包含 `signal` 字段（判断依据）
- ✅ 包含 `solution` 字段（解决方案）
- ✅ 包含 `verify` 字段（验证方式）

**分类标准**：
- **通过**：满足所有评估项 → 继续评估
- **不通过**：缺少关键字段（decision/alternatives/signal）→ 过滤，记录理由

**理由模板**：缺少判断结构，无法抽象为可复用的判断资产

### 2. 判断结构质量（Decision Shape Quality）

评估候选是否包含"如何判断"的结构，而非"做了什么"的步骤。

**评估项**：
- ✅ 能明确回答"Decision being made"（当时在判断什么）
- ✅ 能明确回答"Alternatives rejected"（拒绝了哪些备选方案）
- ✅ 能明确回答"Discriminating signal"（靠什么可观测信号做出分叉）

**分类标准**：
- **通过**：能抽象出判断结构 → 继续评估
- **不通过**：只是步骤描述，无法抽象 → 过滤，记录理由

**理由模板**：只是步骤复现，无法抽象为判断结构

### 3. 可复用性（Reusability）

评估候选是否能在不同场景下复用。

**评估维度**：

**3.1 时间维度（长期价值）**
- **问题**：这条信息在未来（无论是同一项目的后续工作，还是其他项目）还能帮我提前做出正确判断吗？
- **评估标准**：
  
  **跨项目长期价值**（broad scope）：
  - ✅ 能抽象出通用判断模式
  - ✅ 不依赖特定项目/技术栈版本
  - ✅ 包含可迁移的判断依据（signal）
  
  **项目内长期价值**（narrow/medium scope）：
  - ✅ 能在同一项目的后续工作中复用
  - ✅ 包含项目特定的判断依据和约束（如架构决策、技术栈选择、团队约定等）
  - ✅ 即使在项目特定环境下，也能形成可复用的判断模式

**3.2 空间维度（跨项目价值）**
- **问题**：这条判断是否只适用于当前项目的特定约束？
- **评估标准**：
  - ✅ 判断依据（signal）具有通用性
  - ✅ 解决方案（solution）可适用于类似场景
  - ✅ 不依赖项目特定的配置/路径

**3.3 抽象层次**
- **问题**：这条候选能否抽象为更高层次的原则？
- **评估标准**：
  - ✅ 能从具体案例中抽象出判断原则
  - ✅ 能适用于一类问题，而非单个问题
  - ✅ 包含边界条件（什么情况下不适用）

**分类标准**：
- **高可复用性**（平均分 ≥ 0.8，且所有维度 ≥ 0.7）→ 适合 experience（长期资产），Scope = broad（跨项目通用）
- **中等可复用性**（平均分 0.6-0.8，或时间维度 ≥ 0.6）→ 适合 experience（项目级长期资产），Scope = narrow/medium（项目特定或同类问题）
  - **注意**：即使是项目特定的经验，只要能在项目内长期复用，就应该进入 experience，标记为适当的 Scope
- **低可复用性**（平均分 < 0.6，且时间维度 < 0.6）→ 适合 session/worklog（临时记录），或过滤

**理由模板**：
- 高：具有长期价值，能跨项目复用，适合进入 experience（Scope = broad）
- 中：具有项目内长期价值，适合进入 experience（Scope = narrow/medium），即使在项目特定环境下也能形成可复用的判断模式
- 低：过于具体/临时，无复用价值，适合保留为临时记录

**重要**：项目级经验如果能在同一项目的未来复用，即使不能跨项目，也应当被积累到 experience 库中，通过 Scope 字段区分适用范围。

### 4. 沉淀载体适配性（Deposition Target Fitness）

评估候选最适合沉淀到哪个载体。

**载体类型**：

| 载体 | 适用场景 | 评估信号 |
|------|---------|---------|
| **experience**（经验库） | 长期判断资产，需要提醒/指针 | ✅ 高可复用性 + ✅ 判断结构完整 |
| **rules**（规则库） | 可自动判定且高频的约束 | ✅ 可自动检查 + ✅ 高频出现 |
| **skills**（流程能力） | 可复用的流程或重复步骤 | ✅ 流程性质 + ✅ 多次重复 |
| **services**（服务上下文） | 服务边界/配置规范/考古知识 | ✅ 服务特定 + ✅ 长期有效 |
| **session/worklog**（项目记录） | 临时的、无复用价值的记录 | ✅ 低可复用性（时间维度 < 0.6） |

**分类标准**：
- **自动分类**：根据评估结果，推荐最适合的载体（可多选，但优先 ROI 最高的）
- **人最终决定**：推荐结果仅供参考，用户可调整

**输出要求**：
- 推荐载体 + 理由 + 预期收益（下次如何变快/变稳）

## 评估流程

### 阶段 1：自动评估（experience-collector）

**时机**：检测到 EXP-CANDIDATE 时自动执行

**流程**：
1. **结构完整性评估** → 不通过则过滤
2. **判断结构质量评估** → 不通过则过滤
3. **可复用性评估**（初步）→ 记录评估结果和置信度
4. **沉淀载体适配性评估**（初步）→ 记录推荐载体

**输出**：
- 通过所有评估的候选 → 暂存，记录评估结果和推荐载体
- 不通过的候选 → 过滤，记录过滤理由

**特点**：
- 静默处理，不干扰主对话
- 只过滤明显不通过的候选
- 边界模糊的候选仍暂存，标记"需要人工判断"

### 阶段 2：用户选择与确认（experience-depositor）

**时机**：用户执行 `/flow 沉淀 ...` 时

**流程**：
1. **展示候选**：展示暂存的候选，包含评估结果和推荐载体
2. **用户选择**：用户选择要沉淀的候选
3. **可复用性评估（二次确认）**：对用户选择的候选，详细评估可复用性（3 个维度）
4. **沉淀载体适配性评估（二次确认）**：根据详细评估结果，推荐沉淀载体
5. **用户确认载体**：展示推荐载体和理由，用户确认或调整
6. **冲突检测**：检查与现有经验的冲突
7. **治理方案展示**：如果检测到需要治理，展示方案并请求用户确认
8. **写入**：按确认的载体写入

**特点**：
- 用户参与关键决策
- 详细的二次评估，提供充分的判断依据
- 用户可调整推荐载体

## 与旧设计的对比

### 移除的内容

1. **成长过滤器（单独机制）**：合并到"可复用性评估"维度中
2. **重复的评估逻辑**：统一为两阶段评估流程

### 保留的内容

1. **噪音过滤器**：整合为"结构完整性评估"和"判断结构质量评估"
2. **沉淀分流**：整合为"沉淀载体适配性评估"

### 改进的内容

1. **多维度评估**：从单一问题升级为多维度评估
2. **明确分类标准**：每个评估项都有明确的通过/不通过标准
3. **统一实现**：所有评估逻辑集中在候选评估机制中，避免重复
4. **用户体验优化**：用户选择后再进行详细评估，避免选择后又被过滤

## 实现要求

### 1. 统一评估 Skill

创建 `candidate-evaluator` Skill，统一实现所有评估逻辑：

**职责**：
- 结构完整性评估
- 判断结构质量评估
- 可复用性评估（3 个维度）
- 沉淀载体适配性评估

**输入**：EXP-CANDIDATE JSON
**输出**：评估结果（通过/不通过、置信度、推荐载体、理由）

### 2. 评估结果数据结构

```json
{
  "structuralIntegrity": {
    "passed": true,
    "missingFields": [],
    "reason": ""
  },
  "decisionShapeQuality": {
    "passed": true,
    "reason": ""
  },
  "reusability": {
    "temporalDimension": {
      "passed": true,
      "score": 0.9,
      "reason": ""
    },
    "spatialDimension": {
      "passed": true,
      "score": 0.8,
      "reason": ""
    },
    "abstractionLevel": {
      "passed": true,
      "score": 0.85,
      "reason": ""
    },
    "overall": "high" | "medium" | "low",
    "confidence": 0.85
  },
  "depositionTarget": {
    "recommended": ["experience"],
    "alternatives": ["session"],
    "reason": "",
    "expectedBenefit": ""
  },
  "requiresHumanJudgment": false,
  "filterReason": ""
}
```

### 3. 集成点

- **experience-collector**：调用 `candidate-evaluator`，执行阶段 1 评估
- **experience-depositor**：调用 `candidate-evaluator`，执行阶段 2 评估
